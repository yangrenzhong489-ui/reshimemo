# レシメモ 🧾

> レシートをパッと記録して、今月いくら使ったかをすぐ確認できる、シンプルな支出管理アプリです。

## 📱 これは何？

「レシメモ」は、日々の買い物のレシートをもとに支出を手軽に記録できるスマホアプリです。金額・カテゴリ・写真を数タップで入力するだけで、今日・今月の支出とカテゴリ別の内訳がすぐに分かります。

React Native + Expo + TypeScriptで作られた個人開発アプリで、OCRやAIに頼らず「まず手入力でしっかり使えるものを作る」ことを目標にしたMVP（実用最小限の製品）です。

## 💡 解決する課題

- 家計簿アプリは高機能すぎて続かない・入力が面倒で三日坊主になりがち
- レシートを溜め込んでしまい、後でまとめて入力しようとすると挫折する
- 「今月あといくら使えるか」をパッと確認したいだけなのに、アプリを開くたびに手間がかかる

→ レシメモは、**レシートを見ながらその場で数タップ**で記録を終えられることと、**今日・今月の合計をホーム画面で即座に確認できる**ことにこだわりました。

## 🌟 差別化ポイント

- **銀行連携なしで安心**: 口座やクレジットカードには一切接続しません。ID・パスワードや金融情報を預ける必要がなく、レシートを撮る・入力するだけで使えます。
- **レシート写真で支出記録**: レシートを撮影するだけでOCRが金額・日付・店名を自動抽出。数タップで記録が終わります。
- **ムダ買いに気づける**: 今月と先月の支出を自動比較し、増えているカテゴリ・お店を教えてくれる「ムダ買いチェック」機能で、使いすぎに気づきやすくします。
- **節約ミッションで続けやすい**: 「今週は食費を◯◯円以内に」といった、支出データから自動生成される無理のない目標で、家計簿を継続しやすくします。
- **シンプルで日本語UI**: 多機能すぎて挫折しがちな家計簿アプリとは逆に、日本語だけのシンプルな画面構成で家計簿初心者でも迷いません。

## 🧠 工夫した点

- **段階的なMVP開発**: 最初からOCRやAI分類、写真添付に手を広げず、「手入力→ローカル保存→グラフ表示」の順に機能を積み上げて、常に動くアプリを保ちながら開発しました。
- **型安全な設計**: `Expense`・`Category`などのデータ型を早い段階で固め、TypeScriptの型チェックを通して画面間のデータの受け渡しミスを防いでいます。
- **壊れにくいローカル保存**: ストレージの読み書きは全て`try/catch`で保護し、失敗時も例外を投げず安全な値（空配列や`null`）を返す設計にして、アプリが不意にクラッシュしないようにしました。
- **写真ファイルの孤立防止**: 選んだ写真は「保存ボタンを押した瞬間」にだけ端末の永続領域へコピーする設計にし、入力を途中でやめても不要なファイルが残らないようにしています。
- **Safe Area対応の一元管理**: `ScreenContainer`という共通コンポーネントを作り、ステータスバーやホームインジケーターとUIが重ならないよう、全画面で同じロジックを使い回せるようにしました。
- **保存フィードバックの分かりやすさ**: 保存成功時は一瞬「✅ 保存しました」と表示し、金額未入力などのエラーはフィールドの直下に理由を添えて表示するなど、初めて使う人でも迷わないよう配慮しました。
- **共通デザインシステム**: 色は`constants/theme.ts`に一元化（通常操作は青緑系、成功はgreen、注意はamber、危険な操作はred系）し、`AppButton`・`Card`・`EmptyState`という共通コンポーネントを作って、ボタン・カード・空データ表示の見た目を全画面で統一しました。背景は暖かみのあるオフホワイトを基調にしています。
- **AI置き換えを見据えたカテゴリ判定の設計**: カテゴリ判定は「過去の同じ店名の履歴 → キーワードルール」の優先順位で確信度付きの候補を返す関数として独立させ、呼び出し側は最初から非同期・複数候補・確信度・判定理由という形を前提にしています。中身をAI APIに差し替えても呼び出し側の変更が不要な構造です。

## 🏗 セットアップ方法

1. 依存パッケージをインストール

   ```bash
   npm install
   ```

2. （Windows環境でTLS証明書エラーが出る場合）

   環境によっては `npm install` や `npx expo start` の実行時に `UNABLE_TO_VERIFY_LEAF_SIGNATURE` のようなTLSエラーが出ることがあります。その場合は以下のように `NODE_OPTIONS` を設定してから実行してください（OSの証明書ストアを利用するようになります）。

   ```powershell
   $env:NODE_OPTIONS="--use-system-ca"; npm install
   ```

3. （OCR機能を使う場合）レシートOCRはGoogle Cloud Vision APIを利用します。

   1. [Google Cloud Console](https://console.cloud.google.com/)でプロジェクトを作成し、Vision APIを有効化してAPIキーを発行します
   2. 悪用防止のため、発行したAPIキーはGoogle Cloud Console側で「Vision APIのみ利用可」に制限することを推奨します
   3. プロジェクトルートに`.env`ファイルを作成し、`.env.example`を参考に以下を設定します

      ```
      EXPO_PUBLIC_GOOGLE_VISION_API_KEY=あなたのAPIキー
      ```

   > ⚠️ `EXPO_PUBLIC_`が付く環境変数はビルド後のアプリ内に平文で埋め込まれます。個人利用のMVPとしては簡易的にこの方式を使っていますが、公開アプリではAPIキーをサーバー経由で扱うなどの対策を検討してください。APIキーを設定しない場合、OCR機能は「設定が完了していません」というメッセージを表示し、他の機能には影響しません。

## ▶️ 起動方法

開発サーバーを起動します。

```bash
npx expo start
```

（TLS証明書エラーが出る場合は `$env:NODE_OPTIONS="--use-system-ca"; npx expo start`）

起動後、ターミナルに表示されるQRコードを **Expo Go** アプリ（iOS/Android）でスキャンするか、`w`キーでWebブラウザ、`a`キーでAndroidエミュレータ、`i`キーでiOSシミュレータを起動できます。

> ⚠️ **アプリロック（Face ID/Touch ID）機能について**: `expo-notifications`と同様、`expo-local-authentication`もネイティブコードを必要とするため、**Expo Goでは正しく動作しません**。Face ID/Touch ID/指紋認証によるロックの動作を確認するには、`npx expo prebuild`または[EAS Build](https://docs.expo.dev/build/introduction/)で作成した開発ビルド（dev client）を使用してください。Expo Go上ではアプリロックの設定自体は表示されますが、生体認証の可否判定や認証処理が期待通りに動作しない可能性があります。
>
> ⚠️ **Appleでサインインについて**: `expo-apple-authentication`も同様にネイティブコード（Apple Developer側でのSign in with Apple Capability有効化・Entitlement設定）が必要なため、Expo Goでは動作しません。開発ビルドに加えて、Apple Developer Programでのアプリの Capability 設定が別途必要です。また、現時点ではサインインして得られた情報（user id・email・氏名）を端末内に保存するだけで、実際のアカウント同期・サーバー連携は未実装です。

## 🔒 セキュリティ・プライバシー方針

- **銀行口座・クレジットカード連携なし**: 金融機関のログイン情報や口座情報を扱う機能は一切ありません。
- **データはすべて端末内保存**: 支出データ・予算・各種設定は`AsyncStorage`により端末内にのみ保存され、独自サーバーへの送信は行いません。
- **OCR利用時のみ外部送信**: レシートOCR機能を使った場合のみ、撮影した写真がテキスト読み取りのためGoogle Cloud Vision APIに送信されます。OCRを使わなければ写真は端末内にのみ保存されます。
- **APIキー・機密データのGit管理除外**: OCR用APIキーは`.env`で管理し、`.gitignore`でGit管理対象から除外しています（`.env.example`のみコミット）。誤って書き出したバックアップJSON・CSV出力ファイルも`.gitignore`で除外し、レシート写真自体は端末のドキュメント領域に保存されるためリポジトリには含まれません。
- **生体認証（Face ID/Touch ID）はOSが処理**: `expo-local-authentication`はOS標準の認証機構を呼び出すだけで、指紋・顔情報自体はアプリやサーバーに渡らず、常に端末内（Secure Enclave等）で完結します。
- **Appleサインインは現状ローカル保存のみ**: 取得したuser id・email・氏名は端末内にのみ保存し、外部サーバーへは送信していません。
- **通知はローカルスケジュールのみ**: リマインダー・予算超過通知は端末内でスケジュールされるローカル通知で、外部のプッシュ通知サーバーは利用していません。

## 💳 課金実装方針（Plus / Pro）

現状、Plus/Proへの切り替えは`services/billing-service.ts`のスタブ（`purchasePlan()`は常に`{ success: false, reason: 'not-implemented' }`を返す）と`setCurrentPlanForDemo()`によるデモ切り替えのみで、実際の課金処理は未実装です。将来実装する際の方針を以下に整理します。

| 手段 | このアプリでの位置づけ |
|---|---|
| **App Store In-App Purchase（IAP）** | Plus/Proのサブスクリプション課金の本命。ネイティブでのサブスク管理・自動更新・返金/解約のOS標準フローが得られる |
| **RevenueCat** | IAPの実装・レシート検証・複数プラットフォーム対応・契約状態管理を代行してくれるレイヤー。今のFree/Plus/Pro 3段階の構成にそのまま乗せやすい |
| **Apple Pay** | このアプリのPlus/Proには不採用。物理商品・アプリ外決済向けの手段として位置づけ |

- **Apple PayをPlus/Pro課金に使わない理由**: Apple Payは実店舗・ECサイトなど「アプリ外で提供される商品・サービス」向けの決済手段。Plus/Proはアプリ内で消費されるデジタル機能（記録上限解放・OCR回数増・分析機能など）であり、App Storeガイドライン上もデジタルコンテンツ/機能の課金はApple Payではなく**In-App Purchaseの使用が必須**。
- **In-App Purchaseを使う理由**: デジタル機能のサブスク課金として審査上唯一の正攻法であり、サブスクの自動更新・無料トライアル・解約・返金をOS側が処理してくれるため、自前のサーバー決済基盤が不要。
- **RevenueCatを使うメリット**: `StoreKit`の直接実装（レシート検証、Webhook、iOS/Android差異吸収）を肩代わりしてくれ、ダッシュボードで契約状況・MRRなどを可視化できる。個人開発・小規模チームでもIAPを安全かつ短期間で実装しやすい。
- **Apple Payが向いている商品**: このアプリで将来的に「印刷したレシート整理シールの購入」「紙の家計簿ノートとのセット販売」のような**物理商品**や、外部サイトでの単発決済を扱うことになった場合はApple Payが候補になる（現時点でそのような商品計画はない）。
- **将来の課金実装ステップ（案）**:
  1. App Store ConnectでPlus/Proのサブスク商品（月額・年額）を作成
  2. RevenueCatプロジェクトを作成し、上記商品と連携
  3. `services/billing-service.ts`の`purchasePlan()`をRevenueCat SDK呼び出しに置き換え、`getCurrentPlan()`をRevenueCatの契約状態（entitlements）ベースに変更
  4. `setCurrentPlanForDemo()`は開発・QA用のデバッグ機能として残すか、`__DEV__`限定に切り替え
  5. 復元購入（Restore Purchases）とサブスク管理画面（App Store設定への導線）を追加

- `components/` — 画面を構成する部品（`AppButton`・`Card`・`EmptyState`などの共通UI、フォーム、カテゴリ選択、写真ピッカー、OCR読み取り、予算進捗、フィルターUI、設定行、Pro限定機能カード、アプリロック画面、Safe Area対応ラッパーなど）
- `services/` — データ永続化・外部連携（`expense-storage.ts`、`receipt-photo-storage.ts`、`budget-storage.ts`、`csv-export.ts`、`ocr-service.ts`、`category-prediction.ts`、`backup-service.ts`、`notification-service.ts`、`app-lock-service.ts`、`app-lock-settings-storage.ts`、`apple-sign-in-service.ts`、`apple-sign-in-storage.ts`）
- `types/` — 支出データの型定義
- `constants/` — カテゴリ定義（`categories.ts`）、プラン定義（`plans.ts`）、テーマ色（`theme.ts`）
- `utils/` — 日付・金額フォーマット・集計・予算判定・CSV変換ロジックに加え、`waste-detection.ts`（ムダ買いチェック）・`shopping-pattern-analysis.ts`（買い物パターン分析）・`saving-missions.ts`（節約ミッション生成）・`purchase-frequency-ranking.ts`（よく買うものランキング）などの分析ロジック
